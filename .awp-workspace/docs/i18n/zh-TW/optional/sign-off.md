# 簽核（Sign-Off）

**觸發條件**：當需要正式的每任務驗收稽核軌跡時添加此模組——例如：受監管的環境、多方專案中明確的人工批准必須存在記錄，或任何「誰接受了這個以及何時」需要在事後可回答的專案。

## 檔案

`3-verify/SIGN_OFF.md` — 每個已接受任務一個記錄，以只能追加的日誌格式。

## 運作方式

在驗證（Verify）後添加簽核記錄，條件是：
1. 驗收閘門清單已完成
2. 人類審查員做出接受（或條件接受）決策
3. 任何開放的缺口或偏差已在 `GAPS_AND_DEVIATIONS.yaml` 中暫存以供同步（Sync）

每個記錄包含：
- **接受者（Accepted by）** — 做出決策的人類的姓名或角色
- **日期**
- **決策** — `accept`（接受）或 `conditional_accept`（條件接受）
- **條件/備注** — 任何後續義務或已確認的殘留風險
- **暫存於同步的開放項目** — G&D 記錄 ID 或「無」

簽核記錄在事後不得編輯。帶有未解決條件的條件接受，在條件由後續的簽核或同步（Sync）記錄關閉之前，仍保留在記錄中。

## 與迴圈的連接

若 `SIGN_OFF.md` 使用中，記錄一個記錄是在 WORK_QUEUE 中將任務設為 `accepted` 之前的必要步驟。同步（Sync）不直接依賴簽核日誌，但檢查稽核軌跡的審查員將在 DESIGN_INPUTS 和封存記錄旁閱讀它。
