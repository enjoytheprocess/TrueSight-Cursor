# 分層採用（Tiered Adoption）

此範本的設計目標是在任何規模下都能運作——從獨立開發者快速移動，到有正式合規需求的團隊。它擴展的方式是透過**流程深度**：你從一個小的固定核心開始，在需求變得真實時添加選用模組。

沒有可供選擇的層級，也沒有需要符合的框架。流程深度是對你已有多少範本處於活躍狀態的描述，而非你所屬的類別。

---

## 核心

無論流程深度如何，每個使用此範本的專案都運行相同的迴圈：

`設計（Design）→ 建置（Build）→ 驗證（Verify）→ 同步（Sync）`

在該迴圈中，三件事始終存在：

- **建置（Build）↔ 驗證（Verify）返工循環** — 實作在任務被接受之前，在驗證（Verify）中反覆循環
- **缺口和偏差記錄** — 在建置（Build）或驗證（Verify）中發現的任何事情都不會被靜默丟棄；它進入 `3-verify/GAPS_AND_DEVIATIONS.yaml` 並在同步（Sync）時分類
- **IRG 就緒檢查** — 在任何事情進入建置（Build）之前，始終有一個明確的檢查：我知道完成看起來是什麼樣子嗎，我知道我在針對什麼建置嗎？在低流程深度時，這是一個定性判斷；在較高深度時，它成為完整的數值評分標準

這三件事是使此範本在每個規模下都是相同範本的原因。低於此底線，你使用的就是不同的工具了。

---

## 添加模組

核心之外的一切都是你在問題需要時才啟用的模組。觸發條件始終是工作流程中真實的缺口——而非事先制定的計畫。

常見模式：

- 你發現自己失去對進行中工作的追蹤 → 添加**任務佇列**
- 任務觸及安全邊界 → 添加**專家顧問軌道**
- 兩個人同時建置 → 添加**鎖定**
- 你需要誰接受了什麼的記錄 → 添加**簽核**
- 你尚未確定專案方向是否正確 → 添加**構思待辦**

添加模組時無需遷移。建立檔案，開始使用即可。範本的檢查（`make docs-check`）在模組檔案不存在時乾淨地跳過，一旦存在就正確執行。

如果你正在啟動一個專案，已知道你需要某些模組，`make init` 提供預設包，這樣你就不必逐一啟用它們。

---

## 流程深度如何增長

一個專案通常從接近核心的地方開始，隨著範圍、貢獻者或利害關係的增長而累積模組。自然的進程是：

1. **僅核心** — 一個人，一次一個任務，快速移動。迴圈運行；缺口被記錄；IRG 是一個快速的心智檢查。
2. **添加佇列和完整 IRG** — 多個任務同時進行，或第二位貢獻者。定性的 IRG 檢查成為正式的評分標準。TASK_READINESS 成為一個獨立的檔案。
3. **添加可追溯性和回饋** — 實作需要對照規格進行驗證；人類審查員與實作者不同。
4. **添加專家軌道和品質需求** — 領域敏感的工作（安全、API 合約、資料遷移）需要在建置之前和期間進行結構化分析。
5. **添加並發模組** — 多個代理或貢獻者並行工作；鎖定、交接和並發協調層變得必要。

你不需要事先規劃這個進程。每一步都由具體需求觸發。

## 模組技術樹

每個節點是一個模組。箭頭表示「需要」。一旦前置條件就位，可以隨時添加任何模組——你不需要先添加兄弟節點或無關節點。綠色節點需要來自兩個分支的前置條件。

## 模組參考

`已測試（Tested）` = 已在真實專案中端對端演練過。未測試的模組在其文件或範本中可能有只有在實踐中才會浮現的缺口。

| 模組 | 檔案 | 需要 | 何時添加 | 已測試 |
|--------|------|----------|-------------|--------|
| [構思待辦](ideation-backlog.md) | `0-ideation/IDEATION_BACKLOG.yaml` | — | 在承諾設計之前，方向或範圍仍不確定 | 是 |
| [決策記錄](decision-records.md) | `1-design/decisions/` | — | 架構、合約或所有權決策需要持久記錄 | 是 |
| [完整 IRG 評分](full-irg-scoring.md) | `1-design/TASK_READINESS.yaml` | 核心 | 從定性升級至數值評分標準：多個任務同時進行、多個審查員或設計分歧 | 是 |
| [任務佇列](task-queue.md) | `2-build/WORK_QUEUE.yaml` | 核心 | 多個任務同時進行，或階段頻繁中斷 | 是 |
| [專家顧問軌道](specialist-tracks.md) | `3-verify/*_REVIEWS.md` | 完整 IRG + 任務佇列 | 任務觸及安全、實驗、資料遷移、API 合約或事故 | 否 |
| [品質需求](quality-requirements.md) | `1-design/QUALITY_REQUIREMENTS.yaml` | 完整 IRG + 任務佇列 | 非功能或跨領域標準需要每任務執行 | 是 |
| [可追溯性矩陣](traceability-matrix.md) | `3-verify/TRACEABILITY_MATRIX.yaml` | 任務佇列 | 規格/程式碼/測試連結需要維護，且偏移需要可偵測 | 是 |
| [臨時措施](temp-measures.md) | `2-build/TEMP_MEASURES.yaml` | 任務佇列 | 臨時例外或變通方案需要追蹤的移除目標 | 是 |
| [任務依賴](task-dependencies.md) | `2-build/TASK_DEPENDENCIES.yaml` | 任務佇列 | 任務間的上下游關係需要明確追蹤 | 是 |
| [回饋矩陣](feedback-matrix.md) | `3-verify/FEEDBACK_MATRIX.yaml` | 任務佇列 | 人類審查員與實作者不同，且觀察需要結構化記錄 | 是 |
| [簽核](sign-off.md) | `3-verify/SIGN_OFF.md` | 任務佇列 | 需要正式的每任務驗收稽核軌跡 | 否 |
| [發布佇列](release-queue.md) | `4-sync/RELEASE_QUEUE.yaml` | 任務佇列 | 已完成的工作進入部署管道，且發布候選需要追蹤 | 是 |
| [鎖定](locks.md) | `2-build/LOCKS.yaml` | 任務佇列 | 多個代理或貢獻者並行建置 | 否 |
| [交接](handoffs.md) | `4-sync/HANDOFFS.md` | 任務佇列 | 任務所有權在人員或代理之間轉移，且歷史記錄很重要 | 是 |
| [並發協調層](concurrency-overlay.md) | `docs/optional/concurrency-overlay.md` | 鎖定 | 高並發專案需要超越鎖定單獨能提供的完整協調協議 | 否 |
