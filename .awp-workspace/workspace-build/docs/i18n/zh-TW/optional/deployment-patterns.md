# 部署模式（Deployment Patterns）

選擇部署策略和驗證推出安全性的參考指南。

此文件涵蓋*決策*——而非工具設定。將你的實際部署工具連接到你使用的任何平台；在元件 README 中記錄這一點。

## 選擇策略

| 策略 | 何時使用 | 回滾速度 | 風險 |
|---|---|---|---|
| **直接/重啟** | 內部工具、低流量、無狀態 | 快速（重新部署先前版本） | 部署時完全暴露 |
| **功能旗標** | 無基礎設施影響的行為變更 | 即時（切換關閉） | 低——爆炸半徑由旗標範圍控制 |
| **滾動** | 有多個實例的無狀態服務 | 中等（縮回舊版本） | 在推出窗口期間部分暴露 |
| **藍綠** | 具有即時切換的完整環境交換 | 即時（交換回來） | 若環境真正相同則低 |
| **金絲雀** | 高流量或高風險變更，需要指標閘門推出 | 中等（將流量重定向回去） | 低——直到指標通過前，爆炸半徑有限 |

產品行為變更預設使用**功能旗標**，依賴/基礎設施變更使用**滾動**。當變更不可逆、高流量或修改資料時，升級至金絲雀或藍綠。

## 部署前清單

在任何生產部署之前：

- [ ] 回滾程序已記錄並在測試環境中測試過
- [ ] 可觀測性就位：變更的關鍵日誌和指標存在
- [ ] 若推出需要中途停止，功能旗標或流量控制已準備好
- [ ] 資料遷移（若有）已有批准的 `data_migration` 顧問記錄和測試過的回滾路徑
- [ ] 下游消費者未被新版本破壞（合約測試或協調過）
- [ ] 待命或聯絡人知道部署正在進行

## 爆炸半徑控制

在對變更有信心之前限制暴露：

- **縮小旗標範圍**：在擴大之前針對內部使用者或單一地區
- **設置緊急開關**：每個功能旗標必須有記錄的關閉狀態行為
- **限制金絲雀**：預設為 1-5% 流量；不要在沒有指標閘門的情況下自動推進
- **設定時間框架**：定義何時宣布成功或回滾；不要讓「進行中」漂移數天

## 回滾標準

在部署**之前**決定你的回滾觸發條件，而非在事故期間：

- 錯誤率在 `Y` 分鐘內超過 `X%` → 立即回滾
- P99 延遲持續超過 `Xms` 達 `Y` 分鐘 → 回滾
- 任何資料完整性異常 → 回滾並啟動 `incident` 專家軌道
- 業務指標在 `Y` 小時內下降 `X%` → 上報至人類決策

在任務 `Notes` 中記錄這些閾值，或若部署被視為實驗，記錄在 `3-verify/EXPERIMENT_REVIEWS.md`。

## 連接至工作流程

- 在任務 `Notes` 或 `1-design/QUALITY_REQUIREMENTS.md` 中的 `QR-*` 記錄部署策略。
- Schema 或資料變更 → 使用 `advisor_track: data_migration`。
- 指標閘門推出 → 使用 `advisor_track: experiment`。
- 推出期間的生產失敗 → 使用 `advisor_track: incident`。
- 部署後驗證屬於驗證（Verify）階段，應反映在 `3-verify/FEEDBACK_MATRIX.md` 中。
